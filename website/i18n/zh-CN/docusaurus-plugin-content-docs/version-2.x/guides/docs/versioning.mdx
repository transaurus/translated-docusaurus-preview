# 版本管理

您可以使用版本管理CLI基于`docs`目录中的最新内容创建新的文档版本。该特定版本的文档将被保留并可访问，即使`docs`目录中的文档持续更新。

import Tabs from '@theme/Tabs';
import TabItem from '@theme/TabItem';

:::warning

在开始对文档进行版本管理前请三思——这可能会增加贡献者协助改进的难度！

:::

大多数情况下，您并不需要版本管理，因为这只会增加构建时间，并为代码库引入复杂性。版本管理**最适合文档版本间变化频繁且访问量高的网站**。如果您的文档很少变动，请不要为文档添加版本管理。

要更好地理解版本管理的工作原理并判断它是否适合您的需求，您可以继续阅读下文。

## 概述 {#overview}

一个典型的版本化文档站点结构如下：

```bash
website
├── sidebars.json        # sidebar for the current docs version
├── docs                 # docs directory for the current docs version
│   ├── foo
│   │   └── bar.md       # https://mysite.com/docs/next/foo/bar
│   └── hello.md         # https://mysite.com/docs/next/hello
├── versions.json        # file to indicate what versions are available
├── versioned_docs
│   ├── version-1.1.0
│   │   ├── foo
│   │   │   └── bar.md   # https://mysite.com/docs/foo/bar
│   │   └── hello.md
│   └── version-1.0.0
│       ├── foo
│       │   └── bar.md   # https://mysite.com/docs/1.0.0/foo/bar
│       └── hello.md
├── versioned_sidebars
│   ├── version-1.1.0-sidebars.json
│   └── version-1.0.0-sidebars.json
├── docusaurus.config.js
└── package.json
```

`versions.json`文件是按从新到旧排序的版本名称列表。

下表说明了版本化文件如何映射到其版本及生成的URL。

| Path                                    | Version        | URL               |
| --------------------------------------- | -------------- | ----------------- |
| `versioned_docs/version-1.0.0/hello.md` | 1.0.0          | /docs/1.0.0/hello |
| `versioned_docs/version-1.1.0/hello.md` | 1.1.0 (latest) | /docs/hello       |
| `docs/hello.md`                         | current        | /docs/next/hello  |

:::tip

`docs`目录中的文件属于`current`文档版本。

默认情况下，`current`文档版本被标记为`Next`并托管在`/docs/next/*`路径下，但您完全可以配置它以适配项目的发布周期。

:::

当前版本由**文件系统位置**定义，而最新版本由**导航行为**定义。它们可能是同一个版本，也可能不是！（如上表所示的默认配置就将它们视为不同版本：当前版本在`/docs/next`，最新版本在`/docs`。）

## 操作指南 {#tutorials}

### 标记新版本 {#tagging-a-new-version}

1. 首先确保当前文档版本（`./docs`目录）已准备好冻结
2. 输入新版本号

```bash npm2yarn
npm run docusaurus docs:version 1.1.0
```

标记新版本时，文档版本管理机制会执行以下操作：

- 将`docs/`文件夹完整内容复制到新目录`versioned_docs/version-[versionName]/`
- 根据当前[侧边栏](./sidebar/index.mdx)配置生成版本化侧边栏文件（若存在）——保存为`versioned_sidebars/version-[versionName]-sidebars.json`
- 将新版本号追加到`versions.json`

### 创建新文档 {#creating-new-docs}

1. 将新文件放入对应版本目录
2. 根据版本号在对应侧边栏文件中添加新文件的引用

<Tabs>
<TabItem value="Current version structure">

```bash
# The new file.
docs/new.md

# Edit the corresponding sidebar file.
sidebars.js
```

</TabItem>
<TabItem value="Older version structure">

```bash
# The new file.
versioned_docs/version-1.0.0/new.md

```

### 更新现有版本 {#updating-an-existing-version}

您可以同时更新多个文档版本，因为`versioned_docs/`中的每个目录在发布时都对应特定路由。

1. 编辑任意文件
2. 提交并推送更改
3. 变更将发布到对应版本

示例：当您修改`versioned_docs/version-2.6/`中的文件时，只会影响2.6版本的文档。

### 删除现有版本 {#deleting-an-existing-version}

您也可以删除/移除版本。

1. 从`versions.json`中移除该版本

示例：

```diff
[
  "2.0.0",
  "1.9.0",
  // highlight-next-line
- "1.8.0"
]
```

2. 删除版本化文档目录。例如：`versioned_docs/version-1.8.0`。
3. 删除版本化侧边栏文件。例如：`versioned_sidebars/version-1.8.0-sidebars.json`。

## 配置版本化行为 {#configuring-versioning-behavior}

"当前"版本是指`./docs`文件夹对应的版本名称。版本化管理有多种方式，但两种常见模式是：

- 您发布v1后立即开始开发v2（包括其文档）。此时**当前版本**是v2，位于`./docs`源文件夹中，可通过`example.com/docs/next`访问；**最新版本**是v1，位于`./versioned_docs/version-1`源文件夹中，大多数用户通过`example.com/docs`访问。
- 您发布v1后将持续维护一段时间再考虑v2。此时**当前版本**和**最新版本**都指向v1，因为v2文档尚未存在！

Docusaurus的默认配置非常适合第一种场景。我们会将当前版本标记为"next"，您甚至可以选择不发布它。

**针对第二种场景**：如果您发布v1后短期内不计划开发v2，与其创建v1版本导致需要同时维护两个文件夹（`./docs` + `./versioned_docs/version-1.0.0`），可以考虑通过指定路径和标签"伪装"当前版本为已发布版本：

```js title="docusaurus.config.js"
module.exports = {
  presets: [
    '@docusaurus/preset-classic',
    docs: {
      // highlight-start
      lastVersion: 'current',
      versions: {
        current: {
          label: '1.0.0',
          path: '1.0.0',
        },
      },
      // highlight-end
    },
  ],
};
```

这样`./docs`中的文档将通过`/docs/1.0.0`而非`/docs/next`访问，且`1.0.0`会成为导航栏下拉菜单的默认链接版本，您只需维护单个`./docs`文件夹。

我们提供以下插件选项来自定义版本化行为：

- `disableVersioning`：显式禁用版本化功能，即使存在多版本。
- `includeCurrentVersion`：包含当前版本（`./docs`文件夹）的文档。
  - **技巧**：若当前版本是未完成的草稿，可关闭此选项。
- `lastVersion`：设置"最新版本"（`/docs`路由）指向的版本。
  - **技巧**：当当前版本指代持续修补发布的主版本时，`lastVersion: 'current'`是合理配置。最新版本的实际路由基础路径和标签均可配置。
- `onlyIncludeVersions`：定义从`versions.json`中选取部署的版本子集。
  - **技巧**：在开发和部署预览中限制为2-3个版本可提升启动和构建速度。
- `versions`：版本元数据字典。每个版本可配置以下属性：
  - `label`：版本下拉菜单和横幅中显示的标签。
  - `path`：该版本的路由基础路径。默认最新版本使用`/`，当前版本使用`/next`。
  - `banner`：支持`'none'`、`'unreleased'`和`'unmaintained'`。控制每个文档页面顶部显示的横幅。比最新版本更高的版本显示"未发布"，更低的版本显示"不再维护"。
  - `badge`：在该版本文档顶部显示版本号徽章。
  - `className`：为该版本的文档页面`<html>`元素添加自定义类名。

详见[文档插件配置](../../api/plugins/plugin-content-docs.mdx#configuration)获取更多信息。

## 导航栏项目 {#navbar-items}

我们提供多种导航栏项目，助您快速建立导航系统而无需考虑版本化路由问题。

- [`doc`](../../api/themes/theme-configuration.mdx#navbar-doc-link): 指向文档的链接。
- [`docSidebar`](../../api/themes/theme-configuration.mdx#navbar-doc-sidebar): 指向侧边栏首个项目的链接。
- [`docsVersion`](../../api/themes/theme-configuration.mdx#navbar-docs-version): 指向当前查看版本主文档的链接。
- [`docsVersionDropdown`](../../api/themes/theme-configuration.mdx#navbar-docs-version-dropdown): 包含所有可用版本的下拉菜单。

这些链接会按以下顺序寻找合适的版本进行跳转：

1. **当前活动版本**：用户正在浏览的版本（若当前页面由该文档插件提供）。若非文档页面，则回退至...
2. **偏好版本**：用户上次查看的版本。若无历史记录，则回退至...
3. **最新版本**：由`lastVersion`选项配置的默认跳转版本。

## 推荐实践 {#recommended-practices}

### 仅在必要时进行文档版本控制 {#version-your-documentation-only-when-needed}

例如：你正在为npm包`foo`构建文档，当前版本为1.0.0。随后发布了一个修复小错误的补丁版本1.0.1。

是否需要为此创建新的文档版本1.0.1？**通常不需要**。根据语义化版本规范，1.0.1和1.0.0的文档不应存在差异（因为没有新增功能）。为此创建新版本只会产生不必要的重复文件。

### 控制版本数量 {#keep-the-number-of-versions-small}

建议将版本数量控制在10个以下。**极有可能**你会保留大量无人阅读的过时文档版本。例如[Jest](https://jestjs.io/versions)当前版本为`27.4`，但仅维护包括`25.X`在内的几个最新文档版本。保持精简 😊

:::tip[归档旧版本]

若在Jamstack服务商（如[Netlify](../../deployment.mdx)）部署站点，服务商会将每次生产构建保存为不可变URL的快照。可将归档版本作为外部链接指向这些不可变URL。Jest官网和Docusaurus官网均采用此模式来减少主动构建的版本数量。

:::

### 在文档中使用绝对路径导入 {#use-absolute-import-within-the-docs}

避免在文档中使用相对路径导入。因为创建版本时路径会失效（嵌套层级变化等原因）。可使用Docusaurus提供的指向`website`目录的`@site`别名。例如：

```diff
- import Foo from '../src/components/Foo';
+ import Foo from '@site/src/components/Foo';
```

### 通过文件路径链接文档 {#link-docs-by-file-paths}

引用其他文档时使用带`.md`扩展名的相对文件路径，Docusaurus会在构建时将其重写为实际URL路径。文件将正确链接至对应版本。

```md
The [@hello](hello.mdx#paginate) document is great!

See the [Tutorial](../getting-started/tutorial.mdx) for more info.
```

### 全局或版本关联的资产文件 {#global-or-versioned-collocated-assets}

需明确图片等资产文件是按版本隔离还是多版本共享。

若资产需版本化，将其置于文档版本目录并使用相对路径：

```md
![img alt](./myImage.png)

[download this file](./file.pdf)
```

若资产为全局共享，将其放入`/static`并使用绝对路径：

```md
![img alt](/myImage.png)

[download this file](/file.pdf)
```
