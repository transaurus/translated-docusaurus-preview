# 版本管理

您可以使用版本管理CLI基于`docs`目录中的最新内容创建新的文档版本。该特定版本的文档将被保留并可访问，即使`docs`目录中的文档持续更新。

import Tabs from '@theme/Tabs';
import TabItem from '@theme/TabItem';

:::warning

开始对文档进行版本管理前请三思——这可能会增加贡献者协助改进的难度！

:::

大多数情况下您不需要版本管理，这只会增加构建时间并为代码库引入复杂性。版本管理**最适合文档版本间变更频繁且访问量高的网站**。如果您的文档很少变动，请不要为其添加版本管理。

如需了解版本管理机制及判断是否适合您的需求，请继续阅读下文。

## 概述 {#overview}

典型版本化文档站点的结构如下：

```bash
website
├── sidebars.json        # sidebar for the current docs version
├── docs                 # docs directory for the current docs version
│   ├── foo
│   │   └── bar.md       # https://mysite.com/docs/next/foo/bar
│   └── hello.md         # https://mysite.com/docs/next/hello
├── versions.json        # file to indicate what versions are available
├── versioned_docs
│   ├── version-1.1.0
│   │   ├── foo
│   │   │   └── bar.md   # https://mysite.com/docs/foo/bar
│   │   └── hello.md
│   └── version-1.0.0
│       ├── foo
│       │   └── bar.md   # https://mysite.com/docs/1.0.0/foo/bar
│       └── hello.md
├── versioned_sidebars
│   ├── version-1.1.0-sidebars.json
│   └── version-1.0.0-sidebars.json
├── docusaurus.config.js
└── package.json
```

`versions.json`文件是按从新到旧排序的版本名称列表。

下表说明版本化文件如何映射到对应版本及其生成URL。

| Path                                    | Version        | URL               |
| --------------------------------------- | -------------- | ----------------- |
| `versioned_docs/version-1.0.0/hello.md` | 1.0.0          | /docs/1.0.0/hello |
| `versioned_docs/version-1.1.0/hello.md` | 1.1.0 (latest) | /docs/hello       |
| `docs/hello.md`                         | current        | /docs/next/hello  |

:::tip

`docs`目录中的文件属于`current`文档版本。

默认情况下，`current`文档版本被标记为`Next`并托管在`/docs/next/*`路径下，但您完全可以配置其标签和路径以适应项目的发布周期。

:::

## 操作指南 {#tutorials}

### 标记新版本 {#tagging-a-new-version}

1. 首先确保当前文档版本（`./docs`目录）已准备好冻结
2. 输入新版本号

```bash npm2yarn
npm run docusaurus docs:version 1.1.0
```

标记新版本时，文档版本管理机制会执行以下操作：

- 将`docs/`文件夹完整内容复制到新目录`versioned_docs/version-[versionName]/`
- 根据当前[侧边栏](./sidebar/index.mdx)配置生成版本化侧边栏文件（若存在）——保存为`versioned_sidebars/version-[versionName]-sidebars.json`
- 将新版本号追加至`versions.json`

### 创建新文档 {#creating-new-docs}

1. 将新文件放入对应版本目录
2. 根据版本号在相应侧边栏文件中添加新文件引用

<Tabs>
<TabItem value="Current version structure">

```bash
# The new file.
docs/new.md

# Edit the corresponding sidebar file.
sidebars.js
```

</TabItem>
<TabItem value="Older version structure">

```bash
# The new file.
versioned_docs/version-1.0.0/new.md

# Edit the corresponding sidebar file.

```

### 更新现有版本 {#updating-an-existing-version}

您可以同时更新多个文档版本，因为`versioned_docs/`中的每个目录在发布时都对应特定路由。

1. 编辑任意文件
2. 提交并推送变更
3. 变更将发布到对应版本

示例：当您修改`versioned_docs/version-2.6/`中的文件时，只会影响`2.6`版本的文档。

### 删除现有版本 {#deleting-an-existing-version}

您也可以删除/移除版本。

1. 从`versions.json`中移除该版本

示例：

```diff
[
  "2.0.0",
  "1.9.0",
  // highlight-next-line
- "1.8.0"
]
```

2. 删除版本化文档目录。例如：`versioned_docs/version-1.8.0`。
3. 删除版本化侧边栏文件。例如：`versioned_sidebars/version-1.8.0-sidebars.json`。

## 配置版本化行为 {#configuring-versioning-behavior}

"当前"版本是指`./docs`文件夹对应的版本名称。版本化管理有多种方式，但两种常见模式为：

- 发布v1后立即开始开发v2（包括其文档）。此时**当前版本**为v2（位于`./docs`源文件夹），可通过`example.com/docs/next`访问；**最新版本**为v1（位于`./versioned_docs/version-1`源文件夹），大多数用户通过`example.com/docs`访问。
- 发布v1后会长期维护，暂不考虑v2。此时**当前版本**和**最新版本**均为v1，因为v2文档尚未存在！

Docusaurus默认配置非常适合第一种场景。我们会将当前版本标记为"next"，您甚至可以选择不发布该版本。

**针对第二种场景**：如果发布v1后短期内不计划开发v2，与其为v1创建版本化（需同时维护`./docs`和`./versioned_docs/version-1.0.0`两个文件夹），不如通过配置路径和标签"假装"当前版本已是固定版本：

```js title="docusaurus.config.js"
export default {
  presets: [
    '@docusaurus/preset-classic',
    docs: {
      // highlight-start
      lastVersion: 'current',
      versions: {
        current: {
          label: '1.0.0',
          path: '1.0.0',
        },
      },
      // highlight-end
    },
  ],
};
```

这样`./docs`中的文档将通过`/docs/1.0.0`而非`/docs/next`提供服务，`1.0.0`将成为导航栏下拉菜单的默认链接版本，您只需维护单个`./docs`文件夹。

我们提供以下插件选项来自定义版本化行为：

- `disableVersioning`：显式禁用版本化，即使存在多版本也仅包含当前版本。
- `includeCurrentVersion`：是否包含当前版本（`./docs`文件夹）。
  - **技巧**：若当前版本是未完成的草稿，可关闭此选项。
- `lastVersion`：设置"最新版本"（`/docs`路由）指向的版本。
  - **技巧**：当当前版本指代持续发布补丁的主版本时，`lastVersion: 'current'`是合理配置。最新版本的实际路由基础路径和标签均可配置。
- `onlyIncludeVersions`：定义从`versions.json`中部署的版本子集。
  - **技巧**：在开发和部署预览中限制为2-3个版本可提升启动和构建速度。
- `versions`：版本元数据字典。每个版本可配置：
  - `label`：版本下拉菜单和横幅中显示的标签。
  - `path`：该版本的路由基础路径。默认最新版本为`/`，当前版本为`/next`。
  - `banner`：可选`'none'`、`'unreleased'`或`'unmaintained'`，决定每个文档页面顶部显示内容。比最新版本新的版本显示"unreleased"，更旧的显示"unmaintained"。
  - `badge`：在该版本文档顶部显示版本号徽章。
  - `className`：为该版本文档页面的`<html>`元素添加自定义类名。

详见[文档插件配置](../../api/plugins/plugin-content-docs.mdx#configuration)。

## 导航栏项目 {#navbar-items}

我们提供多种导航栏项目，助您快速建立导航而无需考虑版本化路由问题。

- [`doc`](../../api/themes/theme-configuration.mdx#navbar-doc-link): 指向文档的链接。
- [`docSidebar`](../../api/themes/theme-configuration.mdx#navbar-doc-sidebar): 指向侧边栏第一项的链接。
- [`docsVersion`](../../api/themes/theme-configuration.mdx#navbar-docs-version): 指向当前查看版本主文档的链接。
- [`docsVersionDropdown`](../../api/themes/theme-configuration.mdx#navbar-docs-version-dropdown): 包含所有可用版本的下拉菜单。

这些链接会按以下顺序寻找合适的版本进行跳转：

1. **活跃版本**：用户当前正在浏览的版本（若当前页面由该文档插件提供）。若用户不在文档页面，则回退至...
2. **偏好版本**：用户上次查看的版本。若无历史记录，则回退至...
3. **最新版本**：通过`lastVersion`选项配置的默认跳转版本。

## 推荐实践 {#recommended-practices}

### 仅在必要时进行文档版本控制 {#version-your-documentation-only-when-needed}

例如：您正在为npm包`foo`构建文档，当前版本为1.0.0。随后发布了一个修复小错误的补丁版本1.0.1。

是否需要为1.0.1创建新文档版本？**通常不需要**。根据语义化版本规范，1.0.1和1.0.0的文档不应存在差异（因为没有新增功能）。为此创建新版本只会产生不必要的文件冗余。

### 控制版本数量 {#keep-the-number-of-versions-small}

建议将版本数量控制在10个以下。**极有可能**存在大量已无人阅读的过时版本文档。例如[Jest](https://jestjs.io/versions)当前版本为`27.4`，但仅维护包括`25.X`在内的几个最新文档版本。保持精简 😊

:::tip[归档旧版本]

若您的站点部署在Jamstack服务商（如[Netlify](../../deployment.mdx)），服务商会将每次生产构建保存为不可变URL下的快照。您可以将不再更新的归档版本作为外部链接指向这些不可变URL。Jest官网和Docusaurus官网均采用此模式来减少活跃构建的版本数量。

:::

### 在文档中使用绝对导入 {#use-absolute-import-within-the-docs}

避免在文档中使用相对路径导入。因为创建版本时路径层级会发生变化导致失效。可使用Docusaurus提供的指向`website`目录的`@site`别名。示例：

```diff
- import Foo from '../src/components/Foo';
+ import Foo from '@site/src/components/Foo';
```

### 通过文件路径链接文档 {#link-docs-by-file-paths}

引用其他文档时使用带`.md`扩展名的相对文件路径，这样Docusaurus在构建时会将其重写为实际URL路径。文件将自动关联到正确的对应版本。

```md
The [@hello](hello.mdx#paginate) document is great!

See the [Tutorial](../getting-started/tutorial.mdx) for more info.
```

### 全局或版本关联资源 {#global-or-versioned-collocated-assets}

需明确资源（如图片和文件）是按版本隔离还是多版本共享。

若资源需要版本控制，将其置于文档版本目录并使用相对路径：

```md
![img alt](./myImage.png)

[download this file](./file.pdf)
```

若资源是全局的，将其放在`/static`目录并使用绝对路径：

```md
![img alt](/myImage.png)

[download this file](/file.pdf)
```
