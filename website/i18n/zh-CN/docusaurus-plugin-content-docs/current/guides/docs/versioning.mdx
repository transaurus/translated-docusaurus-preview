# 版本管理

您可以使用版本管理CLI基于`docs`目录中的最新内容创建新的文档版本。该特定版本的文档将被保留并可访问，即使`docs`目录中的文档持续更新。

import Tabs from '@theme/Tabs';
import TabItem from '@theme/TabItem';

:::warning

在开始对文档进行版本管理前请三思——这会使贡献者难以协助改进文档！

:::

大多数情况下，您并不需要版本管理功能，因为这只会增加构建时间，并为代码库引入复杂性。版本管理**最适合文档版本间变更频繁且访问量高的网站**。如果您的文档很少变更，请不要为文档添加版本管理功能。

要更好地理解版本管理的工作原理并判断其是否满足您的需求，请继续阅读下文。

## 概述 {#overview}

典型的版本化文档站点结构如下：

```bash
website
├── sidebars.json        # sidebar for the current docs version
├── docs                 # docs directory for the current docs version
│   ├── foo
│   │   └── bar.md       # https://mysite.com/docs/next/foo/bar
│   └── hello.md         # https://mysite.com/docs/next/hello
├── versions.json        # file to indicate what versions are available
├── versioned_docs
│   ├── version-1.1.0
│   │   ├── foo
│   │   │   └── bar.md   # https://mysite.com/docs/foo/bar
│   │   └── hello.md
│   └── version-1.0.0
│       ├── foo
│       │   └── bar.md   # https://mysite.com/docs/1.0.0/foo/bar
│       └── hello.md
├── versioned_sidebars
│   ├── version-1.1.0-sidebars.json
│   └── version-1.0.0-sidebars.json
├── docusaurus.config.js
└── package.json
```

`versions.json`文件是按从新到旧排序的版本名称列表。

下表说明了版本化文件如何映射到其版本及生成的URL。

| Path                                    | Version        | URL               |
| --------------------------------------- | -------------- | ----------------- |
| `versioned_docs/version-1.0.0/hello.md` | 1.0.0          | /docs/1.0.0/hello |
| `versioned_docs/version-1.1.0/hello.md` | 1.1.0 (latest) | /docs/hello       |
| `docs/hello.md`                         | current        | /docs/next/hello  |

:::tip

`docs`目录中的文件属于`current`文档版本。

默认情况下，`current`文档版本被标记为`Next`并托管在`/docs/next/*`路径下，但您完全可以配置它以适配项目的发布周期。

:::

当前版本由**文件系统位置**定义，而最新版本由**导航行为**定义。它们可能是同一个版本，也可能不是！（如上表所示的默认配置就将它们视为不同版本：当前版本在`/docs/next`，最新版本在`/docs`。）

## 操作指南 {#tutorials}

### 标记新版本 {#tagging-a-new-version}

1. 首先确保当前文档版本（`./docs`目录）已准备好冻结
2. 输入新版本号

```bash npm2yarn
npm run docusaurus docs:version 1.1.0
```

标记新版本时，文档版本管理机制将执行以下操作：

- 将`docs/`文件夹完整内容复制到新目录`versioned_docs/version-[versionName]/`
- 根据当前[侧边栏](./sidebar/index.mdx)配置（如果存在）创建版本化侧边栏文件——保存为`versioned_sidebars/version-[versionName]-sidebars.json`
- 将新版本号追加到`versions.json`文件中

### 创建新文档 {#creating-new-docs}

1. 将新文件放入对应版本的文件夹
2. 根据版本号在对应的侧边栏文件中添加新文件的引用

<Tabs>
<TabItem value="Current version structure">

```bash
# The new file.
docs/new.md

# Edit the corresponding sidebar file.
sidebars.js
```

</TabItem>
<TabItem value="Older version structure">

```bash
# The new file.
versioned_docs/version-1.0.0/new.md

# Edit the corresponding sidebar file.
versioned_sidebars/version-1.0.0-sidebars.json
```

</TabItem>
</Tabs>

:::tip

版本化侧边栏文件与标准侧边栏文件类似，都是相对于给定版本的内容根目录——因此对于上面的示例，您的版本化侧边栏文件可能如下所示：

```json
{
  "sidebar": [
    {
      "type": "autogenerated",
      "dirName": "."
    }
  ]
}
```

或手动配置的侧边栏：

```json
{
  "sidebar": [
    {
      "type": "doc",
      "id": "new",
      "label": "New"
    }
  ]
}
```

:::

### 更新现有版本 {#updating-an-existing-version}

您可以同时更新多个文档版本，因为`versioned_docs/`中的每个目录在发布时都对应特定的路由。

1. 编辑任意文件。
2. 提交并推送更改。
3. 变更将发布到对应版本。

示例：当您修改 `versioned_docs/version-2.6/` 目录下的文件时，仅会影响版本 `2.6` 的文档。

### 删除现有版本 {#deleting-an-existing-version}

您也可以删除/移除版本。

1. 从 `versions.json` 中移除目标版本。

示例：

```diff
[
  "2.0.0",
  "1.9.0",
  // highlight-next-line
- "1.8.0"
]
```

2. 删除版本化文档目录。示例：`versioned_docs/version-1.8.0`。
3. 删除版本化侧边栏文件。示例：`versioned_sidebars/version-1.8.0-sidebars.json`。

## 配置版本化行为 {#configuring-versioning-behavior}

"当前"版本是指 `./docs` 文件夹对应的版本名称。版本管理有多种方式，但两种最常见模式是：

- 发布 v1 后立即开始开发 v2（包括其文档）。此时**当前版本**是 v2（存放在 `./docs` 源文件夹），可通过 `example.com/docs/next` 访问；**最新版本**是 v1（存放在 `./versioned_docs/version-1` 源文件夹），大多数用户通过 `example.com/docs` 访问。
- 发布 v1 后将持续维护一段时间再考虑 v2。此时**当前版本**和**最新版本**都指向 v1，因为 v2 文档尚未存在！

Docusaurus 的默认配置非常适合第一种场景。我们会将当前版本标记为 "next"，您甚至可以选择不发布它。

**对于第二种场景**：如果发布 v1 后短期内不计划开发 v2，与其为 v1 创建版本并维护两个文件夹（`./docs` + `./versioned_docs/version-1.0.0`），您可以通过指定路径和标签来"伪装"当前版本为已发布版本：

```js title="docusaurus.config.js"
export default {
  presets: [
    '@docusaurus/preset-classic',
    docs: {
      // highlight-start
      lastVersion: 'current',
      versions: {
        current: {
          label: '1.0.0',
          path: '1.0.0',
        },
      },
      // highlight-end
    },
  ],
};
```

这样 `./docs` 中的文档将通过 `/docs/1.0.0` 而非 `/docs/next` 提供服务，`1.0.0` 将成为导航栏下拉菜单的默认链接版本，您只需维护单个 `./docs` 文件夹。

我们提供以下插件选项来自定义版本化行为：

- `disableVersioning`: 显式禁用版本化，即使存在多个版本。这将使站点仅包含当前版本。
- `includeCurrentVersion`: 包含当前版本（即 `./docs` 文件夹）的文档。
  - **提示**：如果当前版本是开发中的未完成版本，可关闭此选项以避免发布。
- `lastVersion`: 设置"最新版本"（即 `/docs` 路由）指向的版本。
  - **提示**：当当前版本指向持续修补和发布的主版本时，`lastVersion: 'current'` 是合理配置。最新版本的实际路由基础路径和标签均可配置。
- `onlyIncludeVersions`: 定义从 `versions.json` 中部署的版本子集。
  - **提示**：在开发和部署预览中限制为2-3个版本可提升启动和构建速度。
- `versions`: 版本元数据字典。每个版本可配置以下属性：
  - `label`: 版本下拉菜单和横幅中显示的标签。
  - `path`: 该版本的路由基础路径。默认最新版本为 `/`，当前版本为 `/next`。
  - `banner`: 可选 `'none'`、`'unreleased'` 或 `'unmaintained'`。决定每个文档页面顶部显示的内容。比最新版本高的版本标记为"unreleased"，低的标记为"unmaintained"。
  - `badge`: 在该版本文档顶部显示版本名称徽章。
  - `className`: 为该版本文档页面的 `<html>` 元素添加自定义类名。

详见[文档插件配置](../../api/plugins/plugin-content-docs.mdx#configuration)获取更多细节。

## 导航栏项目 {#navbar-items}

我们提供多种文档导航栏项目，帮助您快速建立导航而无需考虑版本化路由。

- [`doc`](../../api/themes/theme-configuration.mdx#navbar-doc-link): 指向单个文档的链接
- [`docSidebar`](../../api/themes/theme-configuration.mdx#navbar-doc-sidebar): 指向侧边栏首项的链接
- [`docsVersion`](../../api/themes/theme-configuration.mdx#navbar-docs-version): 指向当前查看版本主文档的链接
- [`docsVersionDropdown`](../../api/themes/theme-configuration.mdx#navbar-docs-version-dropdown): 包含所有可用版本的下拉菜单

这些链接会按以下顺序寻找合适的版本进行跳转：

1. **活跃版本**：用户当前浏览的版本（如果正在访问该文档插件提供的页面）。若非文档页面，则回退到...
2. **偏好版本**：用户上次查看的版本。若无历史记录，则回退到...
3. **最新版本**：通过 `lastVersion` 选项配置的默认跳转版本。

## `docsVersionDropdown` {#docsVersionDropdown}

默认情况下，[`docsVersionDropdown`](../../api/themes/theme-configuration.mdx#navbar-docs-version-dropdown) 会显示包含所有可用文档版本的下拉菜单。

通过 `versions` 属性可控制显示版本子集及其排序：

```js title="docusaurus.config.js"
export default {
  themeConfig: {
    navbar: {
      items: [
        {
          type: 'docsVersionDropdown',
          // highlight-start
          versions: ['current', '3.0', '2.0'],
          // highlight-end
        },
      ],
    },
  },
};
```

传入 `versions` 对象可覆盖每个版本的显示标签：

```js title="docusaurus.config.js"
export default {
  themeConfig: {
    navbar: {
      items: [
        {
          type: 'docsVersionDropdown',
          // highlight-start
          versions: {
            current: {label: 'Version 4.0'},
            '3.0': {label: 'Version 3.0'},
            '2.0': {label: 'Version 2.0'},
          },
          // highlight-end
        },
      ],
    },
  },
};
```

## 推荐实践 {#recommended-practices}

### 仅在必要时进行文档版本化 {#version-your-documentation-only-when-needed}

例如：您正在为 npm 包 `foo` 构建文档，当前版本为 1.0.0。随后发布了一个小错误修复的补丁版本 1.0.1。

是否需要为此创建新的文档版本 1.0.1？**通常不需要**。根据语义化版本规范，1.0.1 和 1.0.0 的文档不应存在差异（因为没有新功能）。为此创建新版本只会产生不必要的文件重复。

### 控制版本数量 {#keep-the-number-of-versions-small}

作为最佳实践，建议将文档版本数量控制在10个以下。**极有可能**你会发现大量过时的版本化文档根本无人阅读。例如[Jest](https://jestjs.io/versions)当前版本为`27.4`，但仅维护包括最低版本`25.X`在内的几个最新文档版本。保持精简 😊

:::tip[归档旧版本]

如果你在Jamstack提供商（如[Netlify](../../deployment.mdx)）上部署站点，服务商会将每次生产构建作为快照保存在不可变URL下。你可以将不再重建的归档版本作为外部链接指向这些不可变URL。Jest官网和Docusaurus官网都采用这种模式来减少活跃构建的版本数量。

:::

### 在文档中使用绝对路径导入 {#use-absolute-import-within-the-docs}

不要在文档中使用相对路径导入。因为当我们创建新版本时这些路径会失效（嵌套层级变化等原因）。可以使用Docusaurus提供的指向`website`目录的`@site`别名。示例：

```diff
- import Foo from '../src/components/Foo';
+ import Foo from '@site/src/components/Foo';
```

### 通过文件路径链接文档 {#link-docs-by-file-paths}

通过带`.md`扩展名的相对文件路径引用其他文档，这样Docusaurus在构建时能将其重写为实际URL路径。文件将被正确链接到对应版本。

```md
The [@hello](hello.mdx#paginate) document is great!

See the [Tutorial](../getting-started/tutorial.mdx) for more info.
```

### 全局或版本化关联资源 {#global-or-versioned-collocated-assets}

你需要决定如图片等资源是按版本隔离还是跨版本共享。

若资源需要版本化，请将其放入对应版本的docs目录，并使用相对路径：

```md
![img alt](./myImage.png)

[download this file](./file.pdf)
```

若资源是全局性的，请将其放入`/static`目录并使用绝对路径：

```md
![img alt](/myImage.png)

[download this file](/file.pdf)
```
