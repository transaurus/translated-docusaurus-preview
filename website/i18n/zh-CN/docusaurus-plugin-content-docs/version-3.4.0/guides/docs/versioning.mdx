# 版本管理

您可以使用版本管理CLI工具，基于`docs`目录中的最新内容创建新的文档版本。该特定版本的文档将被保留并可访问，即使`docs`目录中的文档持续更新迭代。

import Tabs from '@theme/Tabs';
import TabItem from '@theme/TabItem';

:::warning

开始对文档进行版本管理前请慎重考虑——这可能会增加贡献者协助改进的难度！

:::

大多数情况下您并不需要版本管理功能，这只会增加构建时间并为代码库引入复杂性。版本管理**最适合文档变更频繁且访问量高的网站**。如果您的文档很少更新，请不要为其添加版本管理功能。

如需深入了解版本管理机制并判断是否适合您的需求，请继续阅读下文。

## 概述 {#overview}

典型的版本化文档站点结构如下：

```bash
website
├── sidebars.json        # sidebar for the current docs version
├── docs                 # docs directory for the current docs version
│   ├── foo
│   │   └── bar.md       # https://mysite.com/docs/next/foo/bar
│   └── hello.md         # https://mysite.com/docs/next/hello
├── versions.json        # file to indicate what versions are available
├── versioned_docs
│   ├── version-1.1.0
│   │   ├── foo
│   │   │   └── bar.md   # https://mysite.com/docs/foo/bar
│   │   └── hello.md
│   └── version-1.0.0
│       ├── foo
│       │   └── bar.md   # https://mysite.com/docs/1.0.0/foo/bar
│       └── hello.md
├── versioned_sidebars
│   ├── version-1.1.0-sidebars.json
│   └── version-1.0.0-sidebars.json
├── docusaurus.config.js
└── package.json
```

`versions.json`文件是按从新到旧排序的版本名称列表。

下表说明了版本化文件如何映射到对应版本及生成URL。

| Path                                    | Version        | URL               |
| --------------------------------------- | -------------- | ----------------- |
| `versioned_docs/version-1.0.0/hello.md` | 1.0.0          | /docs/1.0.0/hello |
| `versioned_docs/version-1.1.0/hello.md` | 1.1.0 (latest) | /docs/hello       |
| `docs/hello.md`                         | current        | /docs/next/hello  |

:::tip

`docs`目录中的文件属于`当前`文档版本。

默认情况下，`当前`文档版本被标记为`Next`并托管在`/docs/next/*`路径下，但您可以根据项目发布周期完全自定义这些配置。

:::

## 操作指南 {#tutorials}

### 标记新版本 {#tagging-a-new-version}

1. 首先确保当前文档版本（`./docs`目录）已准备就绪可冻结
2. 输入新版本号

```bash npm2yarn
npm run docusaurus docs:version 1.1.0
```

标记新版本时，文档版本管理机制将执行以下操作：

- 将`docs/`文件夹完整内容复制到新目录`versioned_docs/version-[versionName]/`
- 根据当前[侧边栏](./sidebar/index.mdx)配置生成版本化侧边栏文件（若存在）——保存为`versioned_sidebars/version-[versionName]-sidebars.json`
- 将新版本号追加至`versions.json`

### 创建新文档 {#creating-new-docs}

1. 将新文件放入对应版本目录
2. 根据版本号在对应的侧边栏文件中添加新文件引用

<Tabs>
<TabItem value="Current version structure">

```bash
# The new file.
docs/new.md

# Edit the corresponding sidebar file.
sidebars.js
```

</TabItem>
<TabItem value="Older version structure">

```bash
# The new file.
versioned_docs/version-1.0.0/new.md

# Edit the corresponding sidebar file.
versioned_sidebars/version-1.0.0-sidebars.json
```

</TabItem>
</Tabs>

:::tip

版本化侧边栏文件与标准侧边栏文件类似，都是相对于该版本的内容根目录——因此上例中的版本化侧边栏文件可能如下：

```json
{
  "sidebar": [
    {
      "type": "autogenerated",
      "dirName": "."
    }
  ]
}
```

或手动配置的侧边栏：

```json
{
  "sidebar": [
    {
      "type": "doc",
      "id": "new",
      "label": "New"
    }
  ]
}
```

:::

### 更新现有版本 {#updating-an-existing-version}

您可以同时更新多个文档版本，因为`versioned_docs/`中的每个目录在发布时都对应特定路由。

1. 编辑任意文件。
2. 提交并推送更改。
3. 修改内容将发布至对应版本。

示例：当您修改 `versioned_docs/version-2.6/` 目录下的文件时，仅会影响 2.6 版本的文档。

### 删除现有版本 {#deleting-an-existing-version}

您也可以删除/移除版本。

1. 从 `versions.json` 中移除目标版本。

示例：

```diff
[
  "2.0.0",
  "1.9.0",
  // highlight-next-line
- "1.8.0"
]
```

2. 删除版本化文档目录。示例：`versioned_docs/version-1.8.0`。
3. 删除版本化侧边栏文件。示例：`versioned_sidebars/version-1.8.0-sidebars.json`。

## 配置版本化行为 {#configuring-versioning-behavior}

"当前"版本是 `./docs` 文件夹对应的版本名称。版本管理有多种方式，但两种常见模式为：

- 发布 v1 后立即开始开发 v2（包括其文档）。此时**当前版本**为 v2（位于 `./docs` 源文件夹），可通过 `example.com/docs/next` 访问；**最新版本**为 v1（位于 `./versioned_docs/version-1` 源文件夹），大多数用户通过 `example.com/docs` 访问。
- 发布 v1 后将持续维护一段时间再考虑 v2。此时**当前版本**和**最新版本**均指向 v1，因为 v2 文档尚未存在。

Docusaurus 默认配置完美适配第一种场景。我们会将当前版本标记为"next"，您甚至可以选择不发布它。

**针对第二种场景**：若您发布 v1 且短期内无 v2 开发计划，与其为 v1 创建版本化（需同时维护 `./docs` 和 `./versioned_docs/version-1.0.0` 两个文件夹），不如通过配置路径和标签"伪装"当前版本为已发布版本：

```js title="docusaurus.config.js"
export default {
  presets: [
    '@docusaurus/preset-classic',
    docs: {
      // highlight-start
      lastVersion: 'current',
      versions: {
        current: {
          label: '1.0.0',
          path: '1.0.0',
        },
      },
      // highlight-end
    },
  ],
};
```

这样 `./docs` 中的文档将通过 `/docs/1.0.0` 而非 `/docs/next` 访问，且 `1.0.0` 会成为导航栏下拉菜单的默认链接版本，您只需维护单个 `./docs` 文件夹。

我们提供以下插件选项来自定义版本化行为：

- `disableVersioning`：显式禁用版本化，即使存在多个版本。这将使站点仅包含当前版本。
- `includeCurrentVersion`：包含当前版本（`./docs` 文件夹）的文档。
  - **提示**：如果当前版本是开发中的未发布版本，可关闭此选项。
- `lastVersion`：设置"最新版本"（即 `/docs` 路由）指向的版本。
  - **提示**：当当前版本指代持续修补发布的主版本时，`lastVersion: 'current'` 是合理配置。最新版本的实际路由基础路径和标签均可配置。
- `onlyIncludeVersions`：定义从 `versions.json` 中选取部署的版本子集。
  - **提示**：在开发和部署预览中限制为2-3个版本可提升启动和构建速度。
- `versions`：版本元数据字典。每个版本可配置以下属性：
  - `label`：版本下拉菜单和横幅中显示的标签。
  - `path`：该版本的路由基础路径。默认最新版本为 `/`，当前版本为 `/next`。
  - `banner`：可选 `'none'`、`'unreleased'` 或 `'unmaintained'`。决定每个文档页面顶部显示的内容。高于最新版本的显示"未发布"，低于的显示"不再维护"。
  - `badge`：在该版本文档顶部显示版本号徽章。
  - `className`：为该版本文档页面的 `<html>` 元素添加自定义类名。

详见[文档插件配置](../../api/plugins/plugin-content-docs.mdx#configuration)。

## 导航栏项 {#navbar-items}

我们提供多种导航栏项，助您快速建立导航而无需考虑版本化路由。

- [`doc`](../../api/themes/theme-configuration.mdx#navbar-doc-link)：文档链接。
- [`docSidebar`](../../api/themes/theme-configuration.mdx#navbar-doc-sidebar)：侧边栏首项链接。
- [`docsVersion`](../../api/themes/theme-configuration.mdx#navbar-docs-version)：当前查看版本的主文档链接。
- [`docsVersionDropdown`](../../api/themes/theme-configuration.mdx#navbar-docs-version-dropdown)：包含所有可用版本的下拉菜单。

这些链接会按以下顺序寻找合适的版本：

1. **活跃版本**：用户当前浏览的版本（若处于该文档插件提供的页面）。若非文档页面，则回退至...
2. **偏好版本**：用户最后查看的版本。若无历史记录，则回退至...
3. **最新版本**：通过 `lastVersion` 配置的默认导航版本。

## 推荐实践 {#recommended-practices}

### 仅在需要时进行文档版本化 {#version-your-documentation-only-when-needed}

例如：您正在为 npm 包 `foo` 构建文档，当前版本为 1.0.0。随后发布了一个小错误修复的补丁版本 1.0.1。

是否需要为 1.0.1 创建新文档版本？**通常不需要**。根据语义化版本规范，1.0.1 和 1.0.0 的文档不应有差异（因为没有新功能）。为此创建新版本只会产生不必要的重复文件。

### 控制版本数量 {#keep-the-number-of-versions-small}

建议将版本数量控制在10个以下。**极有可能**存在大量无人阅读的过时版本文档。例如 [Jest](https://jestjs.io/versions) 当前版本为 `27.4`，但仅维护到 `25.X` 的几个最新文档版本。保持精简 😊

:::tip[归档旧版本]

如果您在 Jamstack 提供商（如 [Netlify](../../deployment.mdx)）上部署站点，该提供商会将每次生产构建作为快照保存在不可变 URL 下。您可以将不再重建的归档版本作为外部链接指向这些不可变 URL。Jest 官网和 Docusaurus 官网均采用此模式来减少活跃构建的版本数量。

:::

### 在文档中使用绝对路径导入 {#use-absolute-import-within-the-docs}

不要在文档中使用相对路径导入。因为当我们创建版本时这些路径会失效（嵌套层级变化等原因）。可以使用 Docusaurus 提供的指向 `website` 目录的 `@site` 别名。示例：

```diff
- import Foo from '../src/components/Foo';
+ import Foo from '@site/src/components/Foo';
```

### 通过文件路径链接文档 {#link-docs-by-file-paths}

通过带 `.md` 扩展名的相对文件路径引用其他文档，这样 Docusaurus 能在构建时将其重写为实际 URL 路径。文件将被正确链接到对应版本。

```md
The [@hello](hello.mdx#paginate) document is great!

See the [Tutorial](../getting-started/tutorial.mdx) for more info.
```

### 全局或版本化共置资源 {#global-or-versioned-collocated-assets}

您需要决定资源（如图片和文件）是按版本隔离还是跨版本共享。

若资源需要版本化，请将其放在文档版本目录中，并使用相对路径：

```md
![img alt](./myImage.png)

[download this file](./file.pdf)
```

若资源是全局的，请将其放在 `/static` 目录并使用绝对路径：

```md
![img alt](/myImage.png)

[download this file](/file.pdf)
```
